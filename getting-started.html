Getting started with Ajax Live Search

This quick start guide will guide you through the process of installing ajax live search on your website.

Installing

aLs comes in 2 versions, the lite version and the pro-version. Visit the features page to compare them and check if you need the pro version or the lite version. The lite version if free while the pro version goes for $19.

Installing aLs Lite

To install the light version, log into your WordPress dashboard page and click on the plugins tab then click on add new. On the page that comes up, there is a searchbox that you can use to search for a given plugin. Type in 'Ajax Live Search' and click search. Have a look at the results and click on install when you spot one that has picocodes as the author. WordPress will then automatically install aLs for you and you can start customising it to suite your site.

Alternatively, you can visit the downloads page and download the latest version of aLs; after which you just follow the following steps to install it.

  1. Visit you administrator dashboard and click on the plugins tab;
  2. Click add new then click on upload
  3. Use the form to upload the zip archive that you downloaded from our downloads page.
  4. WordPress will automatically install aLs for you after you click activate.

Installing aLs Pro

If you need extra functionality, then installing aLs pro is the option. Head over to the downloads page and click on the buy button (it costs $19). You will be redirected to the paypal site to finish your payment, and after payment is confirmed; you will receive an email containing the download link.

Click on the link to donwload aLs and then follow the following steps to install.

  1. If you have an active copy of aLs lite, please deactivate it because it shares some of the classes as the pro version.
  2. In your plugins page; click add new then click on upload
  3. Use the form to upload the zip archive that you downloaded from our downloads page.
  4. WordPress will automatically install aLs for you after you click activate.

Searching

You can use any of the following modifiers when searching.

Modifier Description
+ A leading plus sign indicates that this word must be present.
- A leading minus sign indicates that this word must not be present in any of the return rows.
> Indicates that this word should be given more weight.
~ Indicates that this words weight should be negated but not removed. See 2 above.
* When you append an asterisk to a word; we match all words beginning with the value before the asterisk.
"" Returns results that contain the phrase in the parenthesis and in the same order.
author_in: Comma separated Ids of authors whose posts should be searched.
author: or @ The username of an author whose posts should be searched.
author_not_in: Comma separated Ids of authors whose posts should be excluded.
cat: or category_in Comma separated list ids of categories to search
category: Comma, "-" or "+" separated list of category slugs to search in. Use comma to get posts from any of the categories. Use "+" to fetch posts that appear in all categories. Use "-" to exclude posts from the given category.
tagged: Comma or "+" separated list of tag slugs to search in. Use comma to get posts with any of those tags. Use "+" to fetch posts that have all those tags.
post_types: comma separated list of post types to search. The post types given should be the same as those stated in your settings page as searchable.
before: An english date format to use, e.g, last week, 2 months ago, January 1st, 2013 etc. Enclose in quotes. Checkout a list of all available formats here. This query returns posts published before the given date
after: An english date format to use, e.g, last week, 2 months ago, January 1st, 2013 etc. Enclose in quotes. Checkout a list of all available formats here. This query returns posts published after the given date

Below is the algorithym that we use to calculate the relevance of a result.

We take the query and then modify it by stripping words that belong to our list of stopwords and those that appear in more than 50% of the documents. This is because these types of words don't actually add any value to the results.

We then fetch results using our new query and apply the supplied modifiers. If no posts are found; we let WordPress (or your favourite plugin such as relevanssi) do the search and use it's results. This is because they do fuzzy searching by using the 'LIKE' clause hence producing way more result.

For example; a search for "let's all play" won't produce any results in aLs while the default search will probaly return all published posts in the database.

This is because it will search for posts containing either let's, all or play.

In case we fetch any results; we use the following factors to calculate the relevance of each result.

  • IF the query appears in the title, get the tf-idf of the query in respect to the title the multiply that by the weight supplied.
  • IF the query appears in the main content, get the tf-idf of the query in respect to the content the multiply that by the weight supplied.
  • Fetch the weight for each post type the apply it to the found results.
  • Use the current time() to calculate the log of the linear regression from the time the post was published.
  • Calculate the log of the linear regression in respect to the comments count + 1. The +1 is so as to avoid the undefined value of log(0) for posts containing no comments; it also prevents the assignement of 0 to posts with 1 comment since log(1) us 0.
  • Calculate the linear regression of the quality score of the post. To calculate quality; we apply several factors such as click through rate in the search results, word count and our own pagerank algorithym.

To speed up the searching, we calculate some of the data during indexing. The result is a system that works at the same speed as WordPress' inbuilt system (and faster than most of the available options).

If you enable query caching; the time taken to complete a search is almost 2 times less.

NOTE: Some of these features are only available in the premium version.

Customising

aLs comes in a ready to use mode. It is compatible with most themes since it uses the active theme's styles to maintain the theme's look.

However, it comes with it's own search template and also adds it's own styles to the search forms on your website. If your theme does not follow the WordPress theme development guidlines, this might break things a little (in terms of styling).

You should therefore visit the homepage and try to search and see how the suggestions appear. You can also click on one suggestion or click on submit and take a look at how the search page appears.

If you don't like how it feels, or would just like to give it a personal touch, the visit the plugin editor in your dashboard and click on ajax live search; then click on style.css to edit it.

after customising the frontend, head over to the settings tab to customise what appears in your front page; by enabling and disabling certain features.

Enabling additional features

The settings page has 3 tabs; general, searching and result pages.

General settings tab

Currently, the only thing that you can change in the general settings tab is toggling caching on and off.

You can also use this page to index more posts by clicking on the index more tab.

Search settings tab

This tab is used to change how aLs indexes and searches through posts and pages.

To learn more about this features; please visit the features page and take a look at how they work.

Search Results settings tab

Use this tab to tell aLs what and what not to show on the search pages and how to show it.

Adding/Viewing sponsored results.

Ajax search live let's you add sponsored results so that they can be displayed on top of search results when a user submits a relevant query.

In your dashboard, point your cursor to ajax live search and then click on sponsored results. You will see a list of all the sponsored results that you have added. Click on add new to create a new sponsored result. You willl be able to fill a form with the title, description, tags and link of your sponsored result. After saving, your sponsored result will be shown at the top of the search results when someone searches for a relevant query.

You can toggle this on/off in the settings tab. You will also have a chance to specify the number of sponsored results to show in a search query.

Viewing Statistics

aLs logs search queries so that you can analyse them later. You have the option of viewing the statistics as table or in form of graphs. These statistics are especially good when searching for new content ideas.

You can also import the statistics as csv and analyse them offline in your favourite spreadsheet.

If no searches have been made and you need to test it out, just click on the generate dummy data button and aLs will create some dummy content for you to play with.

To view these statistics; in your dashboard, click on the statistics tab of the ajax Live search menu. Here, you can toggle between graphs and tables and also generate dummy data or view it as a pdf document.

Conclusion

By now aLs should be up and running properly. If you need to customise it even further, then make sure to check out the developer's quickstart guide on how to develop an addon for aLs.
